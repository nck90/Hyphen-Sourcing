1) 사업/서비스 개요
1.1 서비스 한 줄 정의
예비 창업가가 상금을 걸고 MVP 제작을 공모하면, 검증된 학생 팀이 24시간 내 작동 MVP를 제출하고, 심사·선정·정산까지 한 번에 처리하는 콘테스트형 해커톤 플랫폼
1.2 핵심 목적(플랫폼 관점)
(창업가) 최소 단가로 빠르게 MVP를 확보하고, 여러 팀의 해석/구현을 비교하여 시장 가능성·제품 방향을 검증


(학생) 실전 과제 기반으로 포트폴리오/협업 경험을 확보하고, 현업 심사/피드백을 통해 성장 및 채용 연결 기회 확대


(운영) 공모-참가-제출-심사-지급-후속관리까지 표준화된 프로세스로 운영 효율 극대화



2) 타겟/이해관계자 정의
2.1 사용자 유형(권한/역할)
의뢰자(창업가/예비창업가)


콘테스트 개설, 브리핑 작성, 상금 예치(에스크로), Q&A/공지, 심사/선정, 상금 승인(지급), 후속 개발 전환(옵션)


참가자(학생 개인)


학생 인증, 포트폴리오 등록, 팀 찾기/참여, 콘테스트 신청, 제출물 업로드


팀(학생 팀)


팀 생성/모집, 역할 분배, 제출물/발표자료 관리, 상금 분배 비율 합의(선택)


심사위원(현업자/VC/기업)


심사 대시보드에서 평가표(Scorecard) 기반 채점, 코멘트 제공, 최종 추천


운영자(Admin/스태프)


학생 인증 승인, 콘테스트 검수, 신고/분쟁 처리, 부정행위 탐지, 정산/세금 처리, 커뮤니티 관리



3) 제품 구조(모듈) — “콘테스트가 중심, 커뮤니티는 부가기능”
3.1 핵심 모듈
콘테스트(해커톤) 모듈: 공모 등록 → 참가/팀 → 24h 진행 → 제출 → 심사 → 우승/정산 → 종료


학생 인증/프로필 모듈: 학생증 인증 + 포트폴리오/스킬/역할


팀 빌딩 모듈: 팀 생성/모집, 매칭 보드, 초대/합류, 팀 잠금


제출/심사 모듈: 제출 폼, 파일/링크, 심사표, 익명 심사, 후보작(쇼트리스트)


정산/에스크로 모듈: 상금 예치, 승인 후 지급, 지연/미선정 처리, 환불/분쟁


커밋틀리(T 시스템) 연동 모듈: 팀별 프로젝트 룸 생성/관리(필수 사용 정책)


커뮤니티 모듈(부가): 게시판/공지/DM, 팟캐스트(라이브) 일정/입장


3.2 참고: 라우드소싱에서 가져올 “콘테스트 UX 핵심”
콘테스트 개설 시 브리핑/옵션/비공개(NDA) 등 선택


심사 단계에서 별점/탈락으로 빠른 선별(참가자에게는 노출되지 않음)


우승 후보작(쇼트리스트) 기능: 후보 선정 후 최종 우승 전 1:1 소통 가능, 후보작 참여 보상(상금의 5%) 옵션 존재


우승작 선정 시 1:1 채팅방 생성 + 상금 승인 버튼으로 산출물 확인 후 지급


연장 정책: 연장 시 상금 상향 요구(동기 유지 장치)


참가자 아이디는 진행/심사 중 비공개, 종료 후 공개 등의 운영 방식



4) 콘테스트(해커톤) 기능 명세 — “라우드소싱급으로 구체화”
아래는 실제 개발팀이 바로 설계/개발로 옮길 수 있도록 화면·데이터·정책 단위로 쪼갠 명세입니다.

4.1 콘테스트 생성(의뢰자) — “개설 마법사(5 Step)”
Step 0. 사전 검수(필수)
의뢰자 본인 인증(휴대폰/이메일)


약관 동의:


제출물 사용권/저작권 정책(우승작만 사용 가능 등)


미선정/연락두절/분쟁 처리 정책


상금 예치(에스크로) 전에는 “임시저장”까지만 가능


라우드소싱도 콘테스트는 총 비용 결제 후 시작 구조


Step 1. 콘테스트 기본 정보
제목(50자), 한 줄 요약(100자)


카테고리:


웹 MVP / 모바일 MVP / 랜딩페이지+대시보드 / AI 기능 PoC / 기타


방식: 온라인 / 오프라인 / 하이브리드


진행시간: 24시간 고정(기본)


시작 일시, 종료 일시(자동 24h)


모집 마감: 시작 24h 전(기본값, 운영자가 설정 가능)


팀 제한: 팀당 2~5명(기본), 개인 참가 허용 여부(옵션)


Step 2. 브리핑(핵심)
라우드소싱이 “브리핑”을 가장 중요하게 다루는 것처럼, 여기서도 브리핑을 템플릿화합니다.
 (라우드소싱도 ‘브리핑 불러오기’ 문서/기능이 존재)
브리핑 폼 항목(권장)
문제 정의(Problem)


타겟 사용자/상황(Who/When)


핵심 유저 시나리오(3개)


MVP 필수 기능(Must-have) / 선택 기능(Nice-to-have)


“작동” 기준(Definition of Done)


예: 로그인/회원가입, 핵심 플로우 1개 완료, DB 연결, 배포 URL 제공


제한 사항


예: 프론트/백 분리 여부, 금지 스택, 외부 API 사용 가능 여부


산출물 요구(필수 체크리스트)


배포 링크(필수)


데모 영상 링크(필수 권장)


Git 저장소(권장/필수 선택)


기본 기획/기능 명세(간단 문서)


심사 기준(가중치 입력)


완성도(30) / 문제적합(25) / UX(20) / 기술구현(15) / 발표(10)


참고자료(첨부)


이미지/PDF/링크


Q&A 가능 시간(선택)


예: 시작 전 1시간, 진행 중 2회(텍스트로만)


Step 3. 상금/비용/옵션
상금(원): 최소 30만원(정책), 5만원 단위 증감(라우드소싱도 5만원 단위 조정 안내)


플랫폼 수수료(초기 MVP 정책 제안)


초기(런칭): 수수료 0~저율(공급/수요 확보 우선)


안정화 후: 라우드소싱처럼 상금 규모에 따라 서비스 이용료 산정(상금 외 별도)


옵션(라우드소싱 벤치마크 기반)


우승 후보작(쇼트리스트) 단계 활성화


후보작 선정 후 1:1 룸을 통해 확정 전 소통 가능


후보작에게 “참여보상” 지급 옵션(예: 총 상금의 5%를 후보작에게 분배)


비공개(시크릿) 콘테스트 + NDA


라우드소싱은 시크릿 콘테스트에서 “회원만 열람 + NDA 동의” 구조를 안내


외부 투표 공유 링크(멘토/후원사 의견 수렴)


라우드소싱도 투표 공유 기능 안내


Step 4. 참가 조건(학생 검증 + 포트폴리오)
학생 인증 “필수”


포트폴리오 “필수”


팀 구성 규칙


팀장 1명 필수


한 사람 1개 팀만 참여(기본)


시작 시각 이후 팀 변경 금지(공정성)


참가자 익명 처리(심사 공정)


라우드소싱은 진행/심사 단계에서 참가자 아이디 비공개 운영을 안내


심사 단계에서는 팀명/개인정보를 숨기고 “Team #”로 표기


Step 5. 결제(상금 예치) 및 게시
결제 수단: 카드/계좌이체(라우드소싱도 결제 방법 문서 존재)


결제 완료 → “모집 시작”


운영자 검수 플래그(초기 운영 권장)


브리핑 품질/불법·유해 내용/사기 위험 검토 후 승인



4.2 탐색/리스트(학생) — “콘테스트 마켓 UI”
리스트 카드 요소(기본)
제목, 상금, 방식(온/오프), 시작 D-day, 참가 요건(학생인증 필요), 팀구성(2~5), 카테고리


“팀 찾는 중” 배지(팀빌딩 활성화 시)


필터/정렬
카테고리, 방식, 상금 구간, 시작일, 남은 시간, 난이도(브리핑 기반 자동 산정)


정렬: 최신/상금높은순/마감임박/조회수



4.3 참가 신청(학생) — “학생증 인증 + 포트폴리오 제출”
(A) 학생 인증 흐름(최소 운영 리스크 버전)
1회성 인증: 계정 단위로 “학생 인증 완료” 상태 부여


업로드 가이드:


주민번호/민감정보는 가리고 업로드(가이드 이미지 제공)


검증 방식:


초기: 운영자 수동 승인(정확도 우선)


확장: AI 판독 + 운영자 샘플링(속도 개선)


데이터 보관:


“이미지 원본”은 검증 완료 후 일정 기간 내 파기


DB에는 “인증상태/학교종류/만료일”만 저장(개인정보 최소화)


(B) 포트폴리오 제출(필수)
필수 필드


GitHub/Behance/Notion/개인사이트 중 1개 이상


프로젝트 1~3개 카드(역할/기여/링크/스크린샷)


선택 필드


기술스택 태그


수상/활동/자격


(C) 콘테스트 신청
개인 참가 또는 팀 참가 선택


팀 참가 시: 팀 선택 → 팀원 모두 “인증 완료” 확인 → 신청 확정


신청 시 자동 생성:


(필수) 팀별 “워크룸(Contest Room)”


(필수) 커밋틀리(T 시스템) 워크스페이스 생성/연동(아래 6장)



4.4 팀 빌딩(Team Formation) — “팀을 만들어주고, 팀을 찾게 해주는 UI”
1) 팀 생성
팀명, 팀 소개(200자), 모집 역할(개발/디자인/PM), 요구 스킬


팀 공개/비공개


팀 초대 링크(만료 가능)


2) 팀 찾기(게시판)
“참가자 카드” 리스트:


역할, 스택, 포트폴리오 대표 링크, 가능 시간, 지역(오프라인일 때)


매칭 기능(초기 MVP)


역할 기반 추천(간단 규칙)


메시지/지원서(짧은 지원서 폼)


“왜 이 팀인가/무엇을 할 수 있나”


3) 팀 잠금(공정성)
해커톤 시작 시각 T0에 팀 구성 자동 잠금


잠금 이후:


멤버 변경 불가


상금 분배 비율 변경 불가(선택 기능이면)



4.5 진행(24h) — “워크룸(Contest Room) 구조”
워크룸은 사실상 라우드소싱의 “콘테스트 페이지 + 문의사항(Q&A) + 공지”를, 해커톤용으로 확장한 형태입니다.
 라우드소싱도 공지/Q&A를 “콘테스트 문의사항” 메뉴로 운영한다고 안내합니다.
워크룸 탭 구성(권장)
브리핑(고정)


공지(운영/의뢰자)


Q&A(공개 질문/답변)


제출(팀별)


중간 체크인(선택: 커밋틀리 연동)


규정/약관


공지/업데이트(운영 측)
Devpost도 참가자 커뮤니케이션을 위해 “Updates” 도구를 별도 운영
 → 동일하게 “업데이트 발행”을 표준 기능으로 두는 것이 운영상 유리합니다.

4.7 심사(Judging) — “별점/탈락 + 쇼트리스트 + 최종선정”
(A) 심사 방식(기본)
심사위원 계정은 “심사 대시보드” 접근


프로젝트별 Scorecard


기준/가중치(브리핑에서 설정)


점수 + 코멘트(의뢰자/팀에게 공개 범위 선택)


(B) 빠른 선별 기능(라우드소싱 벤치마크)
별점/탈락 기능: 의뢰자가 심사 과정에서 작품을 평가·탈락 처리 가능, 이 정보는 참가자에게 노출되지 않음
 → 해커톤에서도 “1차 컷”을 빠르게 하기 위한 UX로 매우 유효


(C) 우승 후보작(쇼트리스트) 단계(옵션, 강력 추천)
라우드소싱은 **‘우승 후보작 선정 옵션’**을 제공하며, 후보작 선정 후 최종 우승작 선정 전 1:1 소통이 가능하다고 안내합니다.

이번 업데이트는 단순히 색상이나 레이아웃을 바꾸는 것이 아니라, "AI가 만든 것 같은" 가벼운 느낌을 전면 배제하고, Linkareer, Loud.kr, SGIC가 지닌 정보 밀도와 묵직한 신뢰감을 결합한 **'초프리미엄 하이엔드 플랫폼'**으로 변모하기 위한 거대한 개편입니다. 아울러 Stitch MCP AI의 디자인 분석을 통해 최적화된 컴포넌트 구조를 도출하여 반영합니다.

12.1 압도적인 프리미엄 디자인 언어 (The "Museum Trust" Aesthetic)
- 완전한 탈(脫) 템플릿: 흔한 Tailwind 기본 섀도우나 과도한 둥근 버튼(border-radius: 9999px) 등 진부한 형태를 버리고, 정제된 스퀘어 형태와 미세한 곡률(radius: 8~12px)을 혼합하여 전문성을 강조합니다.
- 밀도 높은 데이터와 텍스트 계층 구조: Linkareer처럼 한 화면에 많은 정보(해커톤 개요, 상금, 기술 스택, 마감 시간 등)를 담아내면서도, 토스/애플 스타일의 정제된 폰트 자간(-0.015em)과 웨이트 조절을 통해 장시간 조회시에도 눈이 피로하지 않게 설계합니다.
- 시그니처 트러스트 블루 (Trust Blue): SGIC와 Loud.kr이 자아내는 '금융/비즈니스 레벨의 신뢰감'을 확보하기 위해, 흔한 원색 파란색이 아닌 깊이감 있는 딥 네이비 퍼플 계열의 블루(#1a365d ~ #2563eb + 다크 모드)를 시그니처 컬러 기조로 도입합니다.
- 마이크로 인터랙션 (Micro-interactions): 단순히 마우스 호버 시 색이 짙어지는 한계를 넘어, 프레임이 스르륵 떠오르거나(subtle depth increase) 빛이 흘러가는 물리(Physics) 기반의 부드러운 애니메이션(spring frame)을 구현하여 최상의 조작감을 부여합니다.
- Glassmorphism & 비대칭 레이어링: 뻔한 반투명이 아니라 여러 겹의 레이어(Layering)를 입체적으로 쌓아올려, 우승 상금이나 마감일과 같은 핵심 정보가 입체적으로 돌출되어 보이도록 설계합니다.

12.2 핵심 페이지 UI/UX 전면 재설계
(A) 메인 홈 (Hero & Discovery)
- 다층 메인 비주얼: 중앙 집중형 타이틀을 탈피하고, 좌측에는 대담한 메시지와 CTA, 우측에는 역동성 있는 실제 진행 중인 탑 티어 해커톤 카드가 공간감 있게 플로팅되는 비대칭 구조 채택.
- 고밀도 추천 섹션: Loud.kr의 우승작 포트폴리오 섹션처럼, 스크롤을 내릴수록 실시간 제출물, 마감 임박 콘테스트 등이 빈틈없이 짜인 그리드로 쏟아지듯 나타나 활발하게 구동되는 플랫폼임을 어필합니다.

(B) 해커톤 마켓플레이스 (List & Filter)
- Linkareer 스타일 전문 필터링 UX: 좌측에 고정된 아코디언 스타일의 상세 필터(카테고리, 개발 방식, 상금 규모, 직군별)를 배치하고, 우측에는 정보가 가득 찬 리스트형 카드를 스위칭하여 조회할 수 있도록 구성합니다.
- 시각적 상태(Status) 인지도 극대화: '팀 모집 중', '진행 중', '심사 평가 중' 배지를 고대비 색상을 사용하여 배치, 인지 속도와 직관성을 극한으로 끌어올립니다.

(C) 워크룸 및 제출 대시보드 (B2B SaaS Level)
- 프로페셔널 툴의 경험: 기존의 설문 폼 형태를 떠나 Notion이나 Linear와 같이 심플하지만 강력한 인라인 편집창(Inline Editor) 방식의 화면으로 구성. 제출물 작성 시 단계별 진행률을 시각적으로 안내하는 진척도(Progress Indicator) 적용.

12.3 커뮤니티 기능 신설 (지능형 혁신 네트워크)
단순한 텍스트형 게시판을 넘어서, 참가자들이 해커톤 일정이 아니더라도 지속적으로 머무르며 팀을 셋업할 수 있는 '상시 체류형 생태계' 모듈입니다.

(A) 라운지 및 인사이트 피드 (Lounge Feed)
- 소셜 & 마이크로 블로그 구조: 개발 고민, 기술 스택 트렌드, 우승 후기 등을 카드 뉴스와 같이 올릴 수 있는 무한 스크롤 형태의 피드 생태계 구성.
- 코드 스니펫 & 포트폴리오 연동: 게시글 작성 시 자신의 깃허브 코드나 제출 완료한 MVP 산출물을 손쉽게 끌어와 쇼케이스(Showcase)할 수 있는 임베딩 도구 제공.

(B) 상시 팀빌딩 라운지 (Open Team-building Board)
- 해커톤 종속 탈피: 특정 해커톤에 종속된 팀 구인이 아니라, 포트폴리오를 중심으로 한 '상시 팀원 풀(Pool)' 시스템.
- 전문가 매칭 UI: 원티드나 링크드인 방식의 직군/스택 기반 전문 프로필 카드 리스트. 최적화된 미니멀 이력서 카드를 통해 원하는 팀원에게 즉각적인 DM(다이렉트 메시지) 스카우트 요청 가능.

(C) Dev Q&A 및 아카이브 존
- Stack Overflow 스타일 지식인: 해커톤 진행 중 막힌 에러 해결이나 MVP 기획 방향성에 대해 질문하고 답하며 활동 점수(Reputation)를 획득. 이 점수는 추후 해커톤 심사 가산점이나 유저 신뢰 라벨로 전환되는 게이미피케이션(Gamification) 요소 적용.
- 지식 아카이브: 기존 디스코드 기반 라이브 오디오를 보완하여, 종료된 팟캐스트 세션의 요약본 및 VOD/아티클 라이브러리를 구축해 핵심 개발 지식을 영구적으로 자산화.

12.4 실행 프레임워크 (Action Plan with Stitch MCP)
1. Stitch MCP 모델링: 기존 Loud.kr/Linkareer/SGIC 레퍼런스와 텍스트 프롬프트를 배합하여, 가장 진보된 컴포넌트 5종(네비게이션바, 히어로보드, 고밀도 리스트, 커뮤니티 피드, 개발자 프로필 뷰)에 대한 AI 시안을 도출하고 초기 개발 기준점으로 적용.
2. Design System Tokens 재건축: `tailwind.config.ts` 등 전역 시스템 설정 파일 전면 재배치. (컬러 위계 재조정, 타이포그래피 스케일 정밀화, 물리 기반 커스텀 애니메이션 곡선 등록 등)
3. 점진적 무중단 롤아웃: 트래픽 최우선 구역인 랜딩 페이지 및 해커톤 마켓플레이스 화면부터 완벽히 깎아낸 후, 커뮤니티 허브로 연결되는 구조적 UI 마이그레이션을 진행하여 유저 이탈 없는 혁신 모드 달성.
